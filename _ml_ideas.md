# Подбор модели для ML-решений (BARKO)

## 1) Постановка задачи исследования

### 1.1 Описание задачи

Основная ML-задача для MVP - компонент `К3` (первичная оценка и прогноз адаптационных трудностей).  
Нужно по профилю собаки и ответам анкеты оценить риск сложной адаптации и приоритетные зоны внимания.

Дополнительная AI-задача - компонент `К5` (ИИ-чат с опорой на экспертную базу), где важно не "обучить модель с нуля", а обеспечить grounded-ответы и безопасность.

Перспективная ML-задача roadmap - компонент `К6` (автокоррекция плана): на основе динамики ответов пользователя и прогресса выбирать следующий лучший шаг (`next-best-action`) и приоритет действий.

### 1.2 Почему используется ML

- Простые статические правила плохо масштабируются на разные профили собак.
- ML даёт персонализацию вероятности риска, а не только "если-то" сценарии.
- Для УТП BARKO важно показать "прогноз именно для вашей собаки", а не общий совет.

### 1.3 Группа моделей

- Для `К3`: табличные модели (`classical ML`) для классификации риска.
- Для `К5`: `LLM + RAG` на основе готовой внешней модели (без обучения/дообучения своей LLM).
- Для `К6`: модели ранжирования/рекомендаций (`learning-to-rank`, `next-best-action`) в roadmap; в MVP не реализуется.

### 1.4 Критерии выбора модели

1. Качество (`ROC-AUC`, `PR-AUC`, `Recall` high-risk).
2. Интерпретируемость (можно объяснить пользователю и эксперту).
3. Устойчивость на малых/фрагментированных данных.
4. Скорость внедрения в MVP.
5. Потенциал улучшения до собственной модели.

### 1.5 Подбор типов и источников данных

Опираемся на `_data_ideas.md`; детальные описания собранных датасетов и их источников хранятся в каталоге `datasets` (см. файлы `datasets_list.md`).

- C-BARQ supplementary datasets (open subsets из публикаций).
- Dog Aging Project (доступ через заявку).
- Morris Animal Foundation Data Commons.

Ограничение: полного открытого C-BARQ raw-датасета нет, значит baseline строим на объединении доступных подмножеств и аккуратной harmonization-схеме.

## 2) Подбор моделей и обоснование выбора

### 2.1 Сравнительная таблица (К3)

| Кандидат | Плюсы | Риски | Потенциал улучшения | Рекомендация |
|---|---|---|---|---|
| `DummyClassifier` | контрольная нижняя граница | бесполезен продуктово | нет | только для sanity check |
| `LogisticRegression` | интерпретируемая, стабильна на малых данных | слабее на нелинейностях | высокий (feature engineering, calibration) | взять в baseline |
| `DecisionTree` | простая нелинейность, объяснимость | переобучение | средний | взять в baseline |
| `RandomForest` | устойчивее и обычно лучше по качеству | менее прозрачная, тяжелее объяснять | высокий | взять в baseline как основной кандидат |
| `XGBoost/LightGBM` | часто best-in-class на tabular | больше tuning и риск overfit на малых данных | очень высокий | включить как расширенный baseline (если данных достаточно) |

### 2.2 Вывод по К3 (baseline)

- Базовый набор: `LogisticRegression`, `DecisionTree`, `RandomForest`.
- Условно-расширенный набор: `XGBoost/LightGBM`, если объём и качество данных достаточны.
- Текущий рабочий baseline-кандидат для MVP: `RandomForest` (баланс качества и трудозатрат) + обязательная сравнимая интерпретация через `LogisticRegression`.

### 2.3 Вывод по К5 (чат)

- Для `К5` выбираем **готовую LLM через API + RAG** как постоянную стратегию.
- Лучшая стратегия для стартапа: quality control через retrieval, prompt-policy, red-flag rules и human-reviewed knowledge base.
- Фокус развития: качество базы знаний, маршрутизация контекста, контроль безопасности и мониторинг метрик, без обучения собственной LLM.

### 2.4 Решение "своя модель или существующая" по компонентам

| Компонент | Решение на MVP | Решение на следующий этап |
|---|---|---|
| `К3` Прогноз трудностей | существующая baseline-модель (tabular ML) | переход к собственной улучшенной модели после накопления данных |
| `К5` ИИ-чат | существующая LLM API + RAG | улучшение RAG-пайплайна, промптов, safety-правил и экспертной базы |
| `К6` Автокоррекция плана | на этапе MVP отсутствует | разработка собственной модели автокоррекции (next-best-action/ranking) |

## 3) Бенчмарки и метрики для тестирования

### 3.1 Бенчмарки для К3

- Внутренний benchmark: сравнение всех кандидатов на едином split.
- Cross-source benchmark: train на одном источнике, test на другом (где возможно).
- Baseline-to-baseline: сравнение с `DummyClassifier` как нижней границей.

### 3.2 Метрики для К3

- Основная: `ROC-AUC`.
- Обязательные дополнительные:
  - `PR-AUC` (при дисбалансе),
  - `Recall` по классу high-risk,
  - `F1`,
  - `Balanced Accuracy`.
- Калибровка вероятностей:
  - `Brier score`,
  - calibration curve.

Целевой ориентир MVP: `ROC-AUC >= 0.75` на отложенной выборке.

### 3.3 Метрики для К5 (LLM + RAG)

- Retrieval:
  - `Recall@k`,
  - доля ответов с корректными источниками.
- Generation quality:
  - экспертная оценка полезности (Likert 1-5),
  - доля безопасных ответов (без неэтичных рекомендаций),
  - latency (`p50/p95`).
- Product metrics:
  - доля успешно закрытых вопросов,
  - CSAT по чату.

## 5) (Необязательно) Архитектура и пайплайн

- `data/` - сырые и нормализованные данные.
- `src/features/` - сбор и трансформации признаков.
- `src/models/` - обучение, валидация, inference.
- `reports/` - EDA и таблицы бенчмарков.
- `api/` - обёртка `predict_risk(profile)`.

Пайплайн:
`ingest -> clean -> feature_map -> train -> evaluate -> calibrate -> export model -> integrate`.
