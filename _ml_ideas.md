# План разработки baseline-модели (BARKO)

Цель: получить первую рабочую модель для модуля первичной оценки адаптационных трудностей (К3), которую можно проверить на открытых данных и использовать как базу для улучшений.

---

## 0) Входные условия

- Используем план сбора данных из `_data_ideas.md`:
  - open subsets C-BARQ (supplementary data в статьях, например PLOS);
  - Dog Aging Project (доступ по заявке);
  - Morris Animal Foundation Data Commons.
- Ограничение: полного открытого C-BARQ датасета нет, поэтому baseline должен работать на фрагментированных данных.

---

## 1) Формализация ML-задачи

### 1.1 Выбрать первую постановку (MVP)

- Вариант A (самый реалистичный): бинарная классификация риска трудностей адаптации (`низкий/высокий риск`).
- Вариант B: многоклассовая классификация (`низкий/средний/высокий`).
- Вариант C: прогноз по отдельным зонам риска (страх, агрессия, тревога разлуки и т.д.) как multi-label.

Для baseline берем **Вариант A** (проще и устойчивее на малом объеме данных).

### 1.2 Определить таргет

- На основе доступных C-BARQ полей собрать индекс проблемности поведения.
- Зафиксировать правило порога:
  - например, top 30% по интегральному score -> класс `1 (высокий риск)`;
  - остальные -> класс `0`.
- Сделать отдельный markdown с формулой таргета и обоснованием.

---

## 2) Подготовка данных

### 2.1 Data inventory

- Для каждого источника описать:
  - размер выборки;
  - список фич;
  - наличие целевых вопросов/шкал;
  - ограничения лицензии/использования.

### 2.2 Приведение к единой схеме

- Словарь признаков (`feature_map`): как поля из разных наборов мапятся в единые колонки.
- Нормализовать категории (порода, пол, возрастные группы и т.п.).
- Привести шкалы ответов к общему формату.

### 2.3 Очистка

- Удалить дубликаты.
- Обработать пропуски:
  - числовые: median;
  - категориальные: `unknown`.
- Зафиксировать правила фильтрации (какие записи исключаются и почему).

---

## 3) EDA (обязательно до обучения)

- Размер финальной выборки после очистки.
- Доли классов таргета (проверка дисбаланса).
- Распределения ключевых фич.
- Корреляции/связь признаков с таргетом (без leakage).
- Проверка сдвига между источниками (domain shift).

Артефакт: короткий EDA-отчет (таблицы + 5-8 графиков + выводы).

---

## 4) Baseline-модели

### 4.1 Набор baseline-кандидатов

1. `DummyClassifier` (majority class) — контрольная нижняя граница.
2. `LogisticRegression` — интерпретируемый baseline.
3. `DecisionTreeClassifier` — простой нелинейный baseline.
4. `RandomForest` (опционально) — если данных достаточно.

### 4.2 Разделение данных

- Train/valid/test: `70/15/15` или `80/20` + кросс-валидация.
- Стратификация по таргету обязательно.
- Если источников несколько: отдельная проверка generalization (train на одном, test на другом, где возможно).

### 4.3 Метрики

- Основная: `ROC-AUC` (согласовано с вашими требованиями > 0.75 как целевой ориентир).
- Дополнительные: `PR-AUC`, `F1`, `Recall` для класса высокого риска, `Balanced Accuracy`.
- Калибровка вероятностей (Brier score / calibration curve) — желательно.

---

## 5) Интерпретируемость и продуктовая проверка

- Для логистической регрессии: коэффициенты и направление влияния признаков.
- Для деревьев/леса: feature importance + sanity-check с доменной логикой.
- Проверка на "этичность" и здравый смысл:
  - нет очевидно токсичных/дискриминирующих паттернов;
  - рекомендации в продукте не опираются на один сомнительный сигнал.

---

## 6) Критерии готовности baseline

Baseline считаем готовым, если:

- есть воспроизводимый пайплайн подготовки данных;
- есть зафиксированная формула таргета;
- минимум 3 baseline-модели обучены и сравнены;
- метрики посчитаны на отложенной выборке;
- выбран 1 победитель (по качеству + интерпретируемости);
- результаты оформлены в короткий tech note.

---

## 7) Что передать в продукт (после baseline)

- Сериализованный артефакт модели (`.pkl`/`.joblib`).
- JSON/таблица входных фич для backend API.
- Функция `predict_risk(profile) -> risk_score + risk_level`.
- Таблица маппинга в UX:
  - score -> уровень риска;
  - уровень риска -> блок "что делать сейчас" в К3/К4.

---

## 8) План работ по неделям (минимальный)

- Неделя 1: сбор открытых данных + data inventory + схема фич.
- Неделя 2: очистка, таргет, EDA.
- Неделя 3: baseline-обучение, сравнение моделей, выбор победителя.
- Неделя 4: интерпретация, упаковка артефактов, интеграционный черновик для backend.

---

## 9) Риски и запасные варианты

- Мало данных: упрощаем задачу до бинарного риска и усиливаем rule-based слой.
- Сильный дисбаланс: class weights / threshold tuning / PR-AUC как ключевая метрика.
- Слабый transfer между источниками: обучаем source-specific baseline и сравниваем.
- Юридические ограничения данных: используем только datasets с явным разрешением на research/use.
